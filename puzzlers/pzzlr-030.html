Double Trouble

Contributed by Andrew Phillips
Source scala-language mailing list
First tested with Scala version 2.10.0

What is the result of executing the following code?

def printSorted(a: Array[Double]) {
  util.Sorting.stableSort(a)
  println(a.mkString(" "))
}

printSorted(Array(1.0, Double.NaN, 1.1))
printSorted(Array(1.1, Double.NaN, 1.0))
  1. Prints: 
    1.0 1.1 NaN
1.0 1.1 NaN
  2. Prints: 
    1.0 NaN 1.1
1.0 1.1 NaN
  3. Prints: 
    1.0 NaN 1.1
1.1 NaN 1.0
  4. Prints: 
    1.0 NaN 1.1
1.0 NaN 1.1

Explanation

Double extends Ordering, which as a total order requires that a comparison between two values yield either a positive, negative or zero result. However, since 2.10.0 the comparison operations gt, lt and equiv for Double are implemented by the primitive operators >, < and ==. These are compliant with the IEEE floating-point specification, so all comparisons with NaN are false.

This violates the requirements of the ordering function for stableSort. In the first case, it concludes (while sorting the front half of the array) that Double.NaN is not smaller than 1.0, and then that 1.1 is not smaller than 1.0 or Double.NaN, so 1.0 and Double.NaN are left in place.

In the second case, stableSort concludes first that Double.NaN is not smaller than 1.1, then that 1.0 is smaller than 1.1. So 1.0 is inserted into the result array first, followed by 1.1 and Double.NaN, the elements from the front half of the array.

Thus, in the first example, stableSort concludes by direct comparison that Double.NaN is not smaller than 1.0. In the second case, it concludes by transitivity that 1.0 is smaller than Double.NaN, since 1.0 < 1.1 and (Double.NaN < 1.1) == false, i.e. 1.1 <= Double.NaN.
